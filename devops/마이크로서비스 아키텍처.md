# 마이크로서비스 아키텍처
## 정의
- 애플리케이션을 작고 독립적인 서비스 단위로 분리하여 각각 독립적으로 개발, 배포, 운영할 수 있는 아키텍처 스타일.
- 2014년 James Lewis와 Martin Fowler가 개념을 정립함.
## 특징
- 각 서비스는 독립된 프로세스로 실행되며, HTTP API 등을 통해 통신함.
- 서비스는 기능 단위로 나뉘고, 각 기능은 독립적으로 **Deploy** 가능.
- 지속적인 통합과 배포(CI/CD)에 적합하며, 변화에 유연하게 대응할 수 있음.
- DevOps와 밀접한 관련이 있으며, 팀의 자율성과 책임성을 강조함.
## 장점
- 서비스별로 독립적인 개발과 배포 가능 → 빠른 변화 대응.
- 장애 발생 시 전체 시스템이 아닌 해당 서비스에만 영향.
- 다양한 언어나 기술 스택을 서비스별로 선택 가능.
## 단점 및 고려사항
- 서비스 간 통신 복잡도 증가.
- 분산 트랜잭션 처리 어려움.
- 서비스 수 증가에 따른 모니터링, 로그 관리, 배포 복잡도 증가.
- 초기 설계와 인프라 구성이 복잡할 수 있음.
## DevOps와의 관계
- DevOps의 자동화 및 지속적인 통합, 배포 원칙과 잘 부합함.
- 마이크로서비스는 DevOps가 제대로 작동할 수 있는 기반 구조를 제공함.
